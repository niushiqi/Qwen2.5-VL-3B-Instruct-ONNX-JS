# Qwen2.5-VL Python 到 JavaScript 移植方案

## 核心挑战与解决方案

### 1. 库生态差异

**Python 版本依赖：**
- `modelscope`: 模型下载和管理
- `AutoConfig`, `AutoProcessor`, `GenerationConfig`: 自动加载配置和处理器
- `onnxruntime`: ONNX 模型推理
- `numpy`: 数值计算

**JavaScript 解决方案：**
- `onnxruntime-node`: 直接对应 Python 的 onnxruntime
- `@huggingface/transformers`: 提供 AutoTokenizer（但 AutoProcessor 对 Qwen2.5-VL 支持不完整）
- 原生 JavaScript: 替代 numpy 的数值计算

### 2. Processor 处理差异

**问题：**
Python 的 `processor.apply_chat_template()` 会自动：
- 处理图像（resize、normalize、patch extraction）
- 生成 `pixel_values` (5476, 1176)
- 生成 `image_grid_thw` [[1, 74, 74]]
- 生成 `mm_token_type_ids` 标记图像 token 位置

JavaScript 的 `@huggingface/transformers` 对 Qwen2.5-VL 的 processor 支持不完整。

**解决方案：**
使用混合方案：
1. 用 Python 的 processor 预处理一次，保存为 JSON
2. JS 直接加载预处理结果
3. 后续推理完全在 JS 中进行

```javascript
// 加载 Python processor 的输出
const processorOutput = JSON.parse(await fs.readFile('processor_output.json', 'utf-8'));
```

### 3. 数值计算移植

**Python numpy 操作 → JavaScript 原生实现：**

```python
# Python
position_ids = np.zeros((3, batch_size, seq_len), dtype=np.int64)
```

```javascript
// JavaScript
const positionIds = Array(3).fill(0).map(() => 
  Array(batchSize).fill(0).map(() => new Array(seqLen).fill(0))
);
```

**关键函数完全移植：**
- `get_vision_position_ids()`: 计算视觉位置编码
- `get_rope_index()`: 计算 RoPE 位置索引

### 4. ONNX 推理差异

**Python:**
```python
outputs = decoder_session.run(None, dict(
    inputs_embeds=inputs_embeds,
    attention_mask=attention_mask,
    position_ids=position_ids,
    **past_key_values,
))
```

**JavaScript:**
```javascript
const outputs = await decoderSession.run({
  inputs_embeds: inputsEmbeds,
  attention_mask: attentionMaskTensor,
  position_ids: positionIdsTensor,
  ...pastKeyValues
});
```

主要差异：
- JS 需要显式创建 `ort.Tensor` 对象
- JS 使用 `BigInt64Array` 处理 int64 类型
- JS 的 `run()` 是异步的

### 5. 大数组处理优化

**问题：**
```javascript
// 这会导致栈溢出
const logitsData = Array.from(logits.data); // 151936 个元素
const nextToken = lastLogits.indexOf(Math.max(...lastLogits)); // 展开参数过多
```

**解决方案：**
```javascript
// 直接在 TypedArray 上操作，避免转换
let maxVal = -Infinity;
let nextToken = 0;
for (let i = 0; i < vocabSize; i++) {
  const val = logits.data[lastLogitsStart + i];
  if (val > maxVal) {
    maxVal = val;
    nextToken = i;
  }
}
```

### 6. 图像特征注入

**Python:**
```python
inputs_embeds[input_ids == image_token_id] = image_features.reshape(-1, image_features.shape[-1])
```

**JavaScript:**
```javascript
const embedsData = new Float32Array(inputsEmbeds.data);
const featuresData = new Float32Array(imageFeatures.data);
let featureIdx = 0;
for (let i = 0; i < currentInputIds.length; i++) {
  if (currentInputIds[i] === imageTokenId) {
    for (let d = 0; d < embedDim; d++) {
      embedsData[i * embedDim + d] = featuresData[featureIdx * featureDim + d];
    }
    featureIdx++;
  }
}
inputsEmbeds = new ort.Tensor('float32', embedsData, inputsEmbeds.dims);
```

JS 需要手动遍历和复制，因为没有 numpy 的高级索引。

## 性能对比

| 指标 | Python | JavaScript |
|------|--------|------------|
| 图像处理时间 | 27.48s | 23.05s |
| 生成 tokens | 70 | 70 |
| 总耗时 | 67.77s | 64.48s |
| 平均速度 | 1.03 tok/s | 1.09 tok/s |
| 输出内容 | 完全一致 | 完全一致 |

## 最终架构

```
Python Processor (一次性)
    ↓
processor_output.json
    ↓
JavaScript Runtime
    ├── onnxruntime-node (模型推理)
    ├── @huggingface/transformers (tokenizer)
    └── 原生 JS (数值计算)
```

## 关键收获

1. **不要重新发明轮子**：图像预处理逻辑复杂，直接用 Python processor 输出
2. **TypedArray 性能**：大数组操作直接在 TypedArray 上进行，避免转换
3. **逐步调试**：通过 Python 输出对比，快速定位问题
4. **完全等价**：核心算法（RoPE、位置编码）完全移植，保证输出一致
